## 概要

本文聚焦于 AI Agent 场景下的**上下文工程（Context Engineering）**，旨在说明：在多轮推理、工具调用和长链路任务逐渐成为常态的背景下，系统性能的瓶颈，已经不再仅仅来自提示词设计，或模型训练与微调本身，更关键的问题在于，**如何在任务执行的每一个阶段，为模型提供恰当、有效且可控的上下文信息**。

在这一框架下，文章进一步指出，**上下文压缩并不等同于上下文工程本身**，但它确实是其中最核心的问题之一。其基本思路在于：将无需长期保留的信息移出当前上下文窗口，将必须保留的内容压缩为更紧凑的表示，并在后续需要时再按需检索和回填。也就是说，上下文压缩的目标并不是简单地“缩短文本”，而是通过更合理的信息调度，控制上下文规模、提升关键信息密度，并增强系统在长时任务中的稳定性。

本文希望传达的核心观点是：在 AI Agent 时代，真正影响系统稳定性、效率与可扩展性的，不只是提示词本身，而是**上下文的组织与治理方式**。上下文工程正是围绕“哪些信息应当进入上下文、以何种形式进入、何时应当退出、何时应当重新注入”所建立的一整套运行机制，而这也正是构建长时、复杂、可持续运行 Agent 系统的重要基础。

## 一、什么是上下文工程？

### 1. 上下文工程到底是什么？

先说一个最直观的理解：

> 上下文工程，就是围绕当前任务，决定什么信息该给模型看，什么信息先不放进来，什么信息等需要时再拿回来的一整套方法。

如果把提示词工程理解成“怎么跟模型说话”，那上下文工程更像是在解决另一件事：

> **这一轮，到底该把哪些信息摆到模型面前。**

这个问题，放在今天为什么会越来越重要？
因为在很多 AI Agent 场景里，真正拉开效果差距的，往往已经不只是提示词怎么写，也不只是模型够不够强，而是**上下文到底管得好不好**。

现在大家一遇到效果问题，第一反应通常很像：

* 要不要微调一下模型？
* 要不要再训练一点数据？
* 要不要换一个支持更长上下文窗口的模型？

这些方向当然不是没有价值，但很多时候，它们并不是最先该动的那一层。

原因很简单：**它们往往更贵、更慢，而且不一定真的打在问题本身上。**

比如微调这件事，听起来很合理。既然是自己的业务场景，那就拿自己的数据把模型调得更懂业务一点。问题在于，很多 Agent 系统并不是那种边界特别稳定的任务。今天你接一个搜索工具，明天你加一个文件系统，后天又引入新的 MCP 或新的工作流，整个系统的行为方式其实一直在变。

这时候你花很长时间去调模型，很容易遇到一个很现实的问题：

> 模型刚调完，产品形态已经变了。

也就是说，你花了很多力气追模型能力，最后却没有真正追上业务变化本身。

再往深一点说，如果你不是简单微调，而是做更重的训练、后训练，甚至强化学习式的行为优化，那成本会更高，周期也会更长。对大多数做应用、做 Agent、做产品的团队来说，这条路很容易越走越像在“自己再造一层模型能力”。但很多时候，你真正的竞争力，并不在于你能不能重新训练一个模型，而在于你能不能把现有模型用得更好。

那是不是把上下文窗口做大就行了？
这也是一个很容易让人产生错觉的方向。

窗口变大，确实意味着模型“能装下更多东西”，但这不等于模型“更会用这些东西”。

这有点像把办公桌从一米加宽到三米。桌子确实能堆更多文件，但如果没有整理，最后通常不是更高效，而是更乱。上下文也是一样。窗口越大，你越容易把搜索结果、网页正文、日志、代码输出、工具返回一股脑塞进去。最后会发生什么？

* token 花得越来越多
* 推理速度越来越慢
* 无关信息越来越多
* 真正关键的线索反而更容易被埋掉
* 模型虽然“看到了很多”，但不一定真的抓到了最重要的东西

所以问题的关键，从来不只是：

> **能不能放进去。**

真正更关键的是：

> **到底该不该放进去。**

这就是上下文工程真正重要的地方。

在 Agent 系统里，这一点会被放大得特别明显。因为 Agent 不是只调用一次工具就结束了。它会搜索、读取、分析、写入、再搜索、再比对。每往前走一步，都会产生新的 observation。假如这些内容全都原样堆回消息历史里，上下文就会越来越胖，最后变成一个又长又乱、谁都不想再看的包袱。

所以从本质上说，上下文工程做的是一件很朴素、但非常关键的事：

> **帮模型减负，同时帮模型聚焦。**

哪些信息应该常驻，哪些只需要临时注入；
哪些内容应该保留原文，哪些压成摘要就够了；
哪些结果应该移出窗口，哪些要等需要的时候再检索回来。

这些事情处理好了，系统往往会比你急着换更大的模型、堆更长的窗口，甚至立刻去做微调，更快地看到效果。

所以我更愿意把上下文工程理解成这样一层能力：

> 它不是锦上添花的优化项，而是 Agent 真正跑起来之后，最基础的一层运行治理能力。

一个 Agent 能不能长期稳定地工作，很多时候不只是看模型够不够强，而是看你有没有办法持续地把**当前最重要的信息**放到它眼前。

### 2. 上下文工程、提示词工程、记忆工程，到底有什么区别？

这几个词现在经常一起出现，所以也最容易被混着用。
但它们其实解决的，并不是同一个问题。

#### 提示词工程：关注“怎么说”

提示词工程更像是在研究：
怎么把任务说清楚，怎么把要求写明白，怎么让模型更稳定地按照预期输出结果。

它主要关心的通常是这些问题：

* 指令怎么写更清楚
* 角色怎么设定更合适
* 示例怎么给更有效
* 输出格式怎么约束更稳

所以，提示词工程解决的是模型“**如何响应**”的问题，本质上是在优化输入指令层。

#### 记忆工程：关注“什么要长期留下来”

记忆工程关注的则是另一类问题：
哪些信息值得跨轮次保留，哪些结论值得跨任务复用，哪些偏好或状态应该被长期记住。

比如：

* 用户偏好
* 长期任务进度
* 阶段性结论
* 可复用的历史经验

这些都更像是“长期状态”，而不是当前这一轮一定要放进上下文窗口里的东西。

所以，记忆工程解决的是信息“**如何长期保存与再次利用**”的问题。

#### 上下文工程：关注“这一轮该看什么”

上下文工程更偏运行时。
它不只是问“这个信息有没有价值”，而是问：

* 这条信息现在该不该进上下文窗口
* 应该放原文，还是放摘要
* 应该一直保留，还是暂时注入
* 应该给主 Agent 看，还是只给某个子 Agent 看

所以如果用一句更直白的话来区分：

* **提示词工程**，解决的是“怎么写”
* **记忆工程**，解决的是“怎么存”
* **上下文工程**，解决的是“这一轮给什么”

简单整理成表格，大概就是这样：

| 概念    | 主要关注点     |
| ----- | --------- |
| 提示词工程 | 怎么写指令     |
| 记忆工程  | 什么该长期保存   |
| 上下文工程 | 当前窗口里该放什么 |

### 3. 为什么一定要做上下文压缩？

这里先要把一个边界说清楚：

> 上下文压缩，不等于上下文工程本身。
> 更准确地说，它是上下文工程里最核心的问题之一。

因为在 Agent 场景里，上下文几乎一定会越来越长。

一次工具调用，就可能带回来很多内容，比如：

* 搜索结果全文
* 网页正文
* 一大段 JSON
* 程序运行日志
* 文件内容
* 表格内容
* 代码片段

如果这些内容全部原样塞进上下文里，系统很快就会变重。
这时候你就不得不开始思考几个问题：

* 哪些信息其实不需要一直留在窗口里
* 哪些信息可以压缩成更短的形式
* 哪些信息应该等用到的时候再取回来

也正因为这样，上下文压缩才会成为上下文工程里最重要的主题之一。

而且，上下文压缩的目标，并不只是“把文本弄短一点”这么简单。
它真正要解决的是一组更实际的工程问题：

1. 控制 token 成本
2. 控制上下文长度不断膨胀
3. 提高关键信息在上下文里的密度
4. 让模型在长时任务里保持更稳定的推理效果
5. 让多工具、多阶段、长链路任务可以持续跑下去

换句话说，上下文压缩并不是为了让上下文“看起来更短”，而是为了让系统在变复杂之后，依然能把最重要的信息留在最前面，把不重要的信息放到合适的位置上。

这件事看起来像是在“压缩文本”，但本质上其实是在做：

> **信息治理。**

所以，如果要用一句更博客化的话来总结这一节，我会这样说：

> 在 AI Agent 时代，真正决定系统能不能长期跑下去的，不只是模型够不够大，也不只是提示词写得好不好，而是你有没有能力持续地管理好它眼前的上下文。


## 二、上下文工程的常见策略

围绕 Agent 的长上下文问题，业界常见的策略通常可以概括为五类：

1. 上下文卸载（Context Offloading）
2. 上下文缩减（Context Reducing）
3. 上下文检索（Context Retrieving）
4. 上下文隔离（Context Isolation）
5. 上下文缓存（Context Caching）

需要注意的是：

> 这五类策略属于“上下文工程的常见策略”，  
> 并不是说“上下文工程只等于这五项”，  
> 也不是说“上下文压缩只有三部分”。

下面分别来看。

### 1. 上下文卸载：把不必常驻的信息移出上下文窗口

上下文卸载的核心思想是：

> 并不是所有信息都应该长期保留在消息历史中。

有些内容虽然重要，但没有必要一直放在上下文窗口里。  
更合理的做法是把它们保存到外部环境中，例如：

- 文件系统
- 数据库
- Agent 状态对象
- 中间结果存储区

然后在当前上下文中只保留一个简短引用，例如：

- 文件路径
- 文档 ID
- 摘要标签
- 可检索索引

这样做的好处是：

- 完整内容并没有丢
- 当前上下文负担变轻
- 需要时还能重新取回

#### 适合卸载的内容

常见包括：

- 搜索结果全文
- 网页抓取原文
- 超长工具输出
- 日志内容
- 大段表格或 JSON
- 中间分析文档

#### 关键理解

卸载不是“删除”，而是“外部化”。

也就是说，信息从“模型当前直接可见”变成“模型未来可访问”。  
它减少的是上下文窗口的压力，而不是彻底放弃这些信息。

### 2. 上下文缩减：把需要保留的信息压缩成更短形式

如果说卸载是“移出去”，那么缩减就是“留下来，但变短”。

上下文缩减通常包括两种最常见方式：

#### 第一种：摘要

把长内容概括成更短的说明，例如保留：

- 核心结论
- 关键证据
- 重要数据点
- 后续待解决问题

这样模型仍能继续任务，但不必反复读取长原文。

#### 第二种：精简 / 裁剪

并不是所有压缩都要靠自然语言摘要。  
有时候更高效的方式是直接裁掉冗余字段，只保留关键结构。

例如，一个工具输出原本包含：

- 原始全文
- HTML 源码
- 大量 metadata
- 调试信息
- 路径信息

压缩后，可能只需要保留：

- 路径
- 简短说明
- 关键字段
- 可回溯入口

#### 为什么“缩减”不完全等于“摘要”？

因为摘要强调“重新概括”，而缩减强调“让表示更紧凑”。

所以从工程角度讲：

- 摘要是缩减的一种方式
- 裁剪、折叠、结构化保留也属于缩减
- 缩减的目标不是文学性总结，而是降低上下文占用


### 3. 上下文检索：需要时再把相关内容拿回来

当信息被卸载或缩减之后，就必须解决另一个问题：

> 需要用到的时候，怎么重新拿回来？

这就是上下文检索。

检索的核心思想是：

> 不让模型始终背着全部历史，而是在当前任务需要时，只召回最相关的那部分信息。

常见的检索方式大致有两类：

#### 一类：索引 + 语义搜索

适合：

- 文档量大
- 信息分散
- 问题表达和原文措辞不一致
- 需要按语义相关性召回内容

#### 另一类：文件系统 + 简单搜索

适合：

- 文件组织清晰
- 中间产物结构稳定
- 查询目标比较明确
- 希望系统更容易调试和解释

比如通过：

- 文件路径
- 文件名规则
- grep
- glob
- 关键词搜索

来快速定位上下文。

#### 为什么检索也是上下文工程的重要一环？

因为卸载和缩减只是“把东西搬走或压短”，  
而检索负责“在合适的时候重新注入最相关的信息”。

换句话说，真正成熟的上下文工程，不只是会减少信息，还得会动态重建信息。

### 4. 上下文隔离：让不同角色只看自己该看的上下文

上下文隔离的核心思想是：

> 不是所有 Agent、所有子任务，都应该共享同一份上下文。

在复杂系统里，如果一个 Agent 同时负责：

- 规划
- 搜索
- 阅读
- 写作
- 验证
- 工具调用

那么所有中间过程都塞进同一个窗口，最终一定会变得混乱而臃肿。

因此，一个很自然的做法是通过多 Agent 或模块化拆分，让不同角色拥有各自独立的上下文窗口。例如：

- 规划 Agent 只看目标与任务状态
- 搜索 Agent 只看查询与检索结果
- 阅读 Agent 只看材料本身
- 写作 Agent 只看整理后的证据与写作约束

这样做的本质，是通过关注点分离来降低单个上下文窗口的复杂度。

#### 为什么它很重要？

因为很多时候，问题不只是“上下文太长”，而是“上下文里混了太多无关内容”。  
隔离并不直接压缩文本本身，但它能从架构层面减少上下文污染。


### 5. 上下文缓存：避免重复生成、重复检索、重复注入

上下文缓存更像一种系统级优化能力。

它关注的是：

- 某段提示是否已经构造过
- 某份摘要是否已经生成过
- 某次检索结果是否已经拿到过
- 某份中间产物是否已经可复用

如果答案是“已经有了”，那就没有必要每轮都重新生成、重新检索、重新塞回上下文。

#### 它带来的价值包括：

- 减少 token 消耗
- 提高系统速度
- 保持上下文更稳定
- 避免重复信息反复干扰模型

缓存本身不直接等于“压缩”，但它能显著减少不必要的上下文重复注入，因此通常也是上下文工程中非常重要的一环。


## 三、如何从“上下文工程”视角理解上下文压缩？

讲到这里，可以做一个更清楚的总结。

### 1. 上下文工程是总概念

上下文工程解决的是：

> 在任务运行过程中，如何让模型在每一步看到最合适的信息。

它强调的是“上下文的组织、维护、控制与注入”。


### 2. 上下文压缩是其中一个重点问题

上下文压缩更关注的是：

> 当上下文越来越长时，如何控制长度、保留关键信息、减少无效负担。

所以，上下文压缩不是上下文工程的全部，但却是其中最核心的问题之一。


### 3. 如果只看“压缩主线”，可以重点关注三件事

虽然不能把它们说成“上下文工程只有三部分”，但如果专门从“长上下文治理”的视角来看，最关键的主线确实可以概括为三步：

#### 第一步：卸载

把不需要常驻窗口的长内容移出上下文。

#### 第二步：缩减

把必须留在窗口里的内容压缩成更短表示。

#### 第三步：检索

在需要时，再把相关内容按需取回。

所以，更严谨的表达应该是：

> 卸载、缩减、检索，是上下文工程中围绕长上下文控制的一条核心主线。

而隔离与缓存，则更像是帮助这条主线稳定运行的系统级策略。
