## Step 1：放下聊天框

我的第一个结论很直接：如果要把 AI 用到真正重要的编码工作里，就不要再把普通聊天界面当作主要工作方式。

这并不是说聊天产品完全没有价值。相反，在一些场景里它们依然很好用，也确实会偶尔给出让人惊讶的结果。比如我曾经给模型一张界面截图，让它用 SwiftUI 复现，结果出奇地好。

但一旦把这种方式放到真实项目里，尤其是已经存在大量代码、上下文复杂、需要连续推进的项目中，问题就会迅速暴露出来：

- 结果不稳定
- 需要频繁复制代码和命令输出
- 出错后高度依赖人工反复纠正
- 整体效率往往不如自己直接完成

所以，如果想在编码任务里真正获得价值，关键不在聊天框，而在 **agent**。我所说的 agent，至少应该具备几种能力：

- 能读取文件
- 能执行程序
- 能发起 HTTP 请求
- 能在“思考 - 行动 - 观察”的循环中持续推进任务

真正的变化，不是换了一个产品，而是把模型从“问答对象”变成“可以实际干活的系统组件”。

## Step 2：复现我自己的工作

接下来，我开始认真训练自己如何使用 agent。

一开始，体验并不好。输出不稳定、返工很多，看起来并没有比我亲手完成更高效。很多人可能会在这里停下，但我做了一个非常简单、也非常耗时的练习：把我已经手工完成的工作，再让 agent 重新做一遍。

也就是说，我先自己完成一个提交，再让 agent 在看不到我的实现细节的前提下，尽量做出同样的结果。

这当然很痛苦，因为同一件事做了两遍，短期看几乎没有收益。但恰恰是在这个过程中，我才慢慢摸清楚什么样的任务适合 agent，什么样的任务不适合，也开始理解怎样才能让它更稳定地完成工作。

这个阶段我得到的几个关键经验是：

1. 任务必须拆得足够小、足够清晰，不能把一个模糊的大目标直接扔过去。
2. 遇到模糊问题时，最好把规划和执行拆开，不要混在同一轮里。
3. 只要有办法验证结果是否正确，agent 往往就更容易自己修错，成功率也会明显提高。

这一阶段同样让我意识到，知道什么时候不该用 agent，本身也是效率的一部分。

## Step 3：下班前启动 Agent

接下来，我开始尝试一种更实用的日常模式：在下班前最后半小时启动一个或多个 agent，让它们在我不工作的时间里继续推进任务。

思路其实很简单：

- 如果工作时间里还需要一直盯着它，它未必能省时间
- 那就让它去占用那些我本来也不会高效工作的时间
- 用原本空出来的时间，换取第二天一上来就能接手的进展

后来我发现，这种方式特别适合几类任务：

- 深度调研
- 并行探索一些还不确定值不值得继续做的方向
- Issue 和 PR 的初步分流、排查与预审

它尤其擅长使用 `gh` 这类 CLI 工具，所以很适合在 GitHub 相关工作上先做一轮整理。当然，我不会让它直接代替我去回复 issue 或 PR。我更愿意把它当作一个能在隔夜生成初步结果的助手，让我第二天更快进入真正重要的工作。

## Step 4：把稳妥任务外包出去

再往后，我开始更明确地区分什么是适合我自己做的，什么是可以交给 agent 稳定处理的。

重点不是“把所有事都交出去”，而是只把那些成功率高、结果可预期的任务交给它。我的做法通常是：

- 早上先看前一天 agent 生成的结果
- 自己筛一遍，找出适合继续委托的 issue
- 让 agent 在后台处理这些高把握度任务
- 自己则专注于更重要或者更想亲手做的工作

到了这一步，有一件事变得非常重要：**关掉桌面通知。**

真正昂贵的成本不是 agent 在工作，而是它不停打断我。只要它一完成一点东西就把我从当前上下文里拉走，人类这边的效率就会被迅速吞掉。所以中断节奏必须由我来控制，而不是由 agent 来决定。

我也慢慢意识到，把一部分任务交给 agent，并不意味着我自己停止成长。相反，我是在把那些重复、低价值、我不想消耗太多精力的工作外包出去，好把更多时间留给我真正喜欢、也真正值得我投入能力的事情。

## Step 5：工程化 Harness

接下来，重点开始从“如何使用 agent”转向“如何让它更稳定地做对事情”。

如果想让 agent 真正高效，关键不是一次次盯着它纠错，而是给它一套足够好的工作环境，让它能尽快发现自己哪里做错了，并尽量减少重复错误。

我把这种做法叫做 **harness engineering**。

意思很简单：

- 每次 agent 做错一件事，都尽量把这次错误沉淀成机制，避免它以后再犯
- 每次发现某件事原本可以自动验证，就补上验证能力，让它自己确认结果

这通常有两种实现方式。

### 1. 改进隐式提示

第一种方式，是持续完善像 `AGENTS.md` 这样的仓库级说明文件。

如果 agent 经常：

- 运行错误的命令
- 选错 API
- 误解项目约定
- 漏掉必要的检查步骤

那就不要每次都在对话里临时提醒，而是把这些经验直接写进项目说明里，让它以后每次进入仓库时都能继承这些规则。

### 2. 提供真实可执行的工具

第二种方式，是给它提供真正可执行的工具，而不仅仅是文字说明。

例如：

- 截图脚本
- 过滤后的测试命令
- 更聚焦的检查脚本
- 能快速判断结果是否正确的小工具

这些工具的作用，是把原本需要人工盯着检查的事情，变成它可以自己运行、自己验证、自己修正的流程。

## Step 6：让 Agent 始终处于运行中

在做这些工程化改造的同时，我也开始追求另一件事：尽量让 agent 始终处于运行状态。

我会不断问自己一个问题：

**现在有没有什么事情，其实可以交给 agent 去推进？**

这不是为了机械地追求“永远开着 AI”，而是为了培养一种更自然的委派习惯。只要手头有适合委托、并且确实能产生价值的任务，我就会考虑让 agent 在后台先跑起来。

我也发现，这件事和那些更慢、但更仔细的模型配合得很好。它们有时很慢，甚至改一点内容都可能花很久，但换来的往往是更高的完成质量。

与此同时，我并没有特别执着于同时开很多 agent。对我来说，一个后台 agent 往往已经是很好的平衡：

- 我继续做自己深入、手工、而且有乐趣的工作
- 另一个“有点笨但又确实能干活的机器人朋友”在旁边持续推进一些任务

这件事到现在也还没有完全做到。就目前而言，我大概只做到了让后台 agent 覆盖正常工作时间里的 10% 到 20%。但我一直在继续改造自己的工具和工作流，希望把这个比例慢慢提高。

## Today：当前的位置

走到现在，我已经能比较明确地说，现代 AI 工具确实给我的工作带来了真实价值。这种价值不是靠宣传得来的，而是靠反复试错、训练使用方式和改造工作流一点点换来的。

我并不特别在意 AI 最终会不会改变世界，也不想在这件事上站队。对我来说，更重要的是作为一个喜欢构建软件的人，尽可能把眼前这些确实有用的工具用好。

与此同时，我也没有失去警惕。比如我依然担心它会对技能形成带来影响，尤其是对基础还不牢靠的初级开发者来说，这确实可能有代价。我也完全尊重那些选择不用 AI 的人，并不觉得所有人都必须走同样的路。

最后留下来的结论并不是“AI 一定正确”，而是：

- 新工具需要时间去适应
- 真正的收益往往来自工作流改造，而不只是模型本身
- 理性、克制、可验证地使用工具，比情绪化地站队更重要
