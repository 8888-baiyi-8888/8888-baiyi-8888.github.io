## 原文信息

- 标题：*My AI Adoption Journey*
- 作者：Mitchell Hashimoto
- 发布时间：2026 年 2 月 5 日
- 原文链接：<https://mitchellh.com/writing/my-ai-adoption-journey>

## 阅读说明

这篇内容不是逐句直译，而是面向知识库使用场景做的完整中文整理版。它覆盖了原文从 Step 1 到 Step 6 以及 Today 的核心观点、实践路径与阶段变化，适合后续继续补充你的批注、案例和延伸阅读。

## 文章主线

作者把自己采用 AI 工具的过程分成几个阶段。这个过程不是一开始就高效，而是先经历低效、再到勉强可用，最后才逐渐进入真正改变工作流的阶段。整篇文章最有价值的地方，在于它没有把 AI 描述成“立刻提效的魔法”，而是把它视为一种需要训练使用方式、调整任务边界、并持续改造工作环境的工具。

作者的总体态度也很明确：他并不想参与夸张宣传，也不试图证明 AI 的历史地位，只是从一个软件工程师的角度，记录自己如何一步步找到真正有用的工作方法。

## Step 1：放下聊天框

第一步的结论相当激进：不要再尝试通过普通聊天界面完成真正重要的编码工作。

作者并不是说聊天产品毫无价值。相反，他承认聊天界面在日常工作中依然很有用，也确实能在某些场景里给出令人惊艳的结果。比如他曾把一张界面截图贴给模型，让模型用 SwiftUI 复现，结果质量高得超出预期。

但当他把这种方式搬到更多真实项目里时，尤其是已有代码基础、上下文复杂的项目中，问题就出现了：

- 结果经常不稳定
- 需要频繁手动复制代码和命令输出
- 纠错高度依赖人类反复指出问题
- 整体效率甚至不如自己直接做

因此作者认为，如果想在编码任务中真正获得价值，关键不是聊天框，而是 **agent**。在他的定义里，agent 至少需要具备几个能力：

- 读取文件
- 执行程序
- 发起 HTTP 请求
- 在“思考 - 行动 - 观察”的循环中持续推进任务

这一步的核心不是“换个产品”，而是认知上的转变：从把模型当作问答对象，转向把它当作可执行工作的系统组件。

## Step 2：复现你自己的工作

第二步是作者真正开始形成方法论的阶段。

他一开始也试用了 agent 产品，但体验并不理想。输出质量不稳定，返工很多，看起来并没有比自己亲手完成更高效。很多人在这里就会放弃，但作者没有直接下结论，而是做了一个非常“笨但有效”的练习：把自己已经手工完成的工作，再让 agent 重新做一遍。

也就是说，他会先手动完成某个提交，然后再让 agent 在看不到手工答案的前提下，尽可能产出同样质量和功能的结果。

这个过程很痛苦，因为它让同一份工作做了两遍，短期内显然不划算。但这也正是训练期最有价值的地方。通过这种方式，他逐渐从第一性原理上发现了几个关键经验：

1. 任务要拆分成清晰、可执行的小步骤，不要指望一次会话直接完成一整个模糊的大目标。
2. 遇到模糊需求时，最好把“规划”和“执行”分成不同阶段，而不是混在同一轮指令里。
3. 只要给 agent 一个能验证结果是否正确的方法，它就很可能自己修错、减少回归，并显著提高成功率。

这一阶段的重要收获，不只是“怎么让 agent 做事”，还包括“什么时候不该用 agent”。作者强调，知道哪些任务大概率会失败，本身就是效率的一部分。避免把时间浪费在不适合 agent 的问题上，和学会正确使用它同样重要。

到了这一步，他已经开始愿意把 agent 纳入自己的工作流，但还没有明显感受到净效率提升，因为他仍然需要大量盯着 agent 做事。

## Step 3：下班前启动 Agent

为了进一步寻找效率空间，作者开始尝试一种新的日常模式：每天临近下班时，专门留出最后 30 分钟启动一个或多个 agent，让它们在自己不工作的时间里继续推进任务。

这个策略背后的思路很简单：

- 既然工作时间内还需要盯着 agent，未必能省时间
- 那就试着把 agent 放到自己本来就不会高效工作的时段
- 用“原本空出来的时间”换取第二天的前置进展

一开始这件事同样显得麻烦，但作者很快找到了一些特别适合这种模式的任务类型：

- 深度调研任务
- 并行探索一些方向模糊、但值得先探路的想法
- Issue 和 PR 的分流、排查与预审

其中他尤其提到，agent 很擅长使用 `gh` 这类 CLI 工具，因此非常适合帮忙做 GitHub 上的初步梳理。但他不会让 agent 直接代替自己回复 issue 或 PR，而是把它们当作一种“隔夜报告生成器”，帮助自己第二天更快聚焦到高价值任务。

这一步的价值在于：即使 agent 没有完全替代你的工作，它也可以成为第二天开始工作的“热启动器”。尤其在一天的后半段，人往往已经疲劳、注意力下降，这时用剩余精力去布置 agent，比硬撑着继续做高认知负载任务更划算。

## Step 4：把稳妥任务外包出去

到了第四步，作者已经对 agent 擅长什么、不擅长什么有了比较稳定的判断。于是策略也进一步升级：把那些成功率很高、结果大致可预期的任务，直接交给 agent 在后台处理，而自己把精力放到更重要或更喜欢的任务上。

这里的关键不是“让 agent 替代所有事情”，而是明确挑选那些它几乎肯定能做得不错的任务。作者的做法是：

- 早上先看前一天 agent 生成的排查结果
- 自己筛一遍，挑出适合委托的 issue
- 让 agent 在后台持续处理这些高把握度任务
- 自己则进入正常的深度工作状态，处理别的事情

这一阶段里，作者特别强调了一点：**关掉 agent 的桌面通知。**

因为真正昂贵的不是 agent 工作本身，而是频繁的上下文切换。如果 agent 一完成一点事就打断你，人类这边的效率会被严重破坏。所以控制中断节奏必须由人来掌握，而不是由 agent 决定。

这一阶段还有一个很有意思的观点：把部分任务交给 agent，并不意味着你就不再成长。因为你完全可以把那些重复、低价值、你不想投入太多精力的任务外包出去，同时把更多时间投入到自己真正在意、也真正能提升能力的工作上。

到了这里，作者已经进入“很难再回到没有 agent 的状态”的阶段。原因不只是效率，而是工作体验本身发生了变化。他可以把更多思考和创造力留给自己真正喜欢的任务。

## Step 5：工程化 Harness

第五步是整篇文章中最具“系统建设”意味的一步。

作者认为，agent 越能在第一次就做对，或者只需要极少修补就完成任务，效率就越高。而最稳妥的办法，不是单纯寄希望于模型更聪明，而是给它足够快、足够好的工具，让它可以自动知道自己做错了什么。

作者把这套思路称为 **harness engineering**。它表达的是一种习惯：

- 每当 agent 犯了一个错误，就想办法建设机制，避免它以后再犯同类错误
- 每当某件正确行为其实可以被自动验证，就想办法补上验证能力，让 agent 能自己确认

作者提到，这件事通常有两种实现方式：

### 1. 更好的隐式提示

也就是持续完善像 `AGENTS.md` 这样的仓库级说明文件。

如果 agent 经常：

- 执行错命令
- 找错 API
- 误解目录结构
- 漏掉某些必须的检查步骤

那就不要每次在对话里重复提醒，而是把这些规则写进项目说明里，让它以后每次进入仓库都能带着这些约束开始工作。

### 2. 提供真实可执行的工具

仅靠文字提示还不够。很多时候，更有效的是直接给 agent 提供工具，例如：

- 截图脚本
- 过滤后的测试命令
- 针对性的校验脚本
- 可自动判断结果是否正确的小工具

这些工具会把“需要人盯着检查”的工作，尽量转成“agent 可以自己验证”的流程。这样一来，它不仅更容易第一次做对，也更容易在出错后自行修复。

这一步标志着作者的重心从“如何使用 agent”转向“如何建设适合 agent 工作的环境”。这说明 agent 不再只是一个偶尔调用的助手，而开始成为需要工程化支持的长期生产力组件。

## Step 6：让 Agent 始终处于运行中

在推进第五步的同时，作者也在尝试另一个目标：尽量让 agent 始终处于运行状态。

他会经常问自己一个问题：

**现在有没有什么事情，其实可以让 agent 帮我推进？**

这里的重点不是机械地追求“永远开着 AI”，而是持续培养一种委派意识。只要手头存在一项适合委托、且确实能帮到自己的任务，就考虑让 agent 在后台跑起来。

作者也提到，他喜欢把这个目标与较慢但更仔细的模型配合起来。某些模型虽然耗时长，甚至修改一点点内容都可能花三十分钟以上，但代价换来的是结果质量通常更高。

与此同时，他并没有执着于同时运行很多 agent。当前对他来说，一个后台 agent 已经是比较好的平衡点：

- 一边保持自己做深入、手工、且有乐趣的工作
- 一边让那个“有点笨但又神秘地有生产力的机器人朋友”持续推进一些任务

作者也坦言，这个目标还没有完全达成。以当下的状态来看，他大概只能让后台 agent 覆盖正常工作日里的 10% 到 20% 时间。但他正在主动改造自己的工具和工作流，希望未来能持续提高这个比例。

## Today：当前立场

最后，作者总结了自己现在的位置。

经过前面这些阶段后，他已经走到了一个比较稳定的状态：他确实从现代 AI 工具里得到了价值，而且这种价值不是靠幻想或宣传建立起来的，而是建立在现实的试错、训练和工作流改造之上。

他并不执着于讨论 AI 最终会不会“改变世界”，也没有强烈的立场包袱。对他来说，更重要的是作为一个热爱构建软件的人，尽可能用好眼前能帮助自己创造价值的工具。

他也保留了清醒的一面。例如，他明确提到自己对技能形成问题仍然有担忧，特别是对基础还不牢靠的初级开发者而言，这确实可能带来负面影响。他也尊重那些选择不用 AI 的人，并不试图说服所有人接受同样的做法。

整篇文章最后留下来的感觉，不是“AI 一定正确”，而是：

- 新工具需要时间去适应
- 真正的价值往往来自工作流改造，而不只是模型能力本身
- 理性、克制、可验证地使用工具，比情绪化站队更重要

## 我的理解

这篇文章最值得借鉴的，不是某一个产品或某一个 prompt，而是一种循序渐进的 adoption 路线：

1. 先放弃低效用法，不再把聊天框当成主工作台。
2. 用刻意练习的方式摸清 agent 的能力边界。
3. 先从“你不在工作时它替你推进任务”开始建立收益。
4. 再把高确定性的任务稳定外包出去。
5. 最后通过说明文件、脚本和验证链路，把 agent 纳入正式工程体系。

如果把它迁移到自己的仓库里，最先值得做的三件事通常是：

1. 写清楚仓库级规则，例如目录约定、常用命令、测试入口和提交规范。
2. 为高频检查动作补上脚本，让 agent 能自己验证，而不是每次等人来指出问题。
3. 主动识别哪些任务适合委派，哪些任务仍然更适合自己手做。

## 一句话总结

《My AI Adoption Journey》真正写的不是“怎么用 AI 写代码”，而是“一个工程师如何一步步把 AI 从低效玩具，变成值得纳入工作流的生产工具”。
